# ETIC — 易经八卦占卜 iOS 应用设计文档

> ETIC = **E**asy (易) **T**ool-based **I** **C**hing。
> 一款基于中国传统术数(易经 / 八卦 / 六爻 / 梅花易数)的 iOS 占卜应用。

---

## 0. 设计总纲(最重要的一条原则)

**确定性的术数计算与大模型的自然语言解读必须严格分层、互不越界。**

- **起卦、排盘、取象、生克旺衰、定吉凶要素** → 全部由 **预设工具(确定性引擎 + 结构化知识库)** 完成。这一层是纯函数、可单元测试、可离线运行、结果可复现。
- **解读、断语、多轮对话、个性化表达** → 由 **大模型(LLM)** 完成。LLM 只能基于引擎输出的结构化盘面进行解读,**不得自行"计算"卦象、不得篡改盘面数据**。

这样设计的根本原因:大模型在干支纳甲、五行生克、世应旬空这类规则性计算上极不可靠(会幻觉);而传统排盘恰恰是 100% 规则化、可程序化的。把"算"交给代码、把"说"交给模型,才能同时获得**准确性**与**自然交流体验**。

## 1. 产品定位与范围

### 1.1 目标用户
- 对易经 / 传统命理感兴趣的中文用户(主力)。
- 需求场景:针对**具体事项**问占(事业、求财、感情、考试、失物、出行、健康趋势等),以及日常娱乐(每日一卦)。

### 1.2 占卜方法选型

| 方法 | 起卦方式 | 解读复杂度 | 是否纳入 |
| --- | --- | --- | --- |
| **六爻(纳甲筮法)** | 铜钱摇卦 / 时间 / 报数 | 高(装卦、世应、六亲、六神、旬空、生克旺衰) | **MVP 主线** |
| **梅花易数** | 数字 / 时间起卦,体用生克 | 中 | V2 |
| **大衍揲蓍 / 文王课** | 揲蓍法 | 作为六爻的起卦变体 | V2 |
| **每日一卦 / 卦象百科** | — | 低(纯展示) | MVP 辅助 |

> **结论**:MVP 以**六爻**为核心,因为它信息量最大、最适合"具体事项占断",也最能体现"工具计算 + LLM 解读"的分工价值。

### 1.3 合规与免责(国内重点)
- 所有解读结果标注「**传统文化 / 娱乐参考,非科学预测**」。
- 不对医疗、法律、财务、人身安全等高风险领域做绝对化承诺。
- 若主要面向国内,LLM 优先选用**已备案的国产合规模型**,并接入内容安全审核。

## 3. 预设工具层(核心地基)

把它实现为一个**独立、纯函数式、100% 可单元测试**的 Swift Package(`DivinationEngine`)。输入起卦参数,输出一份完整的结构化盘面 JSON。该 JSON **同时**驱动 UI 可视化与作为 LLM 输入。

### 3.1 静态数据资产(随包内置,离线)

- **八卦基础**:乾兑离震巽坎艮坤 —— 卦象(三爻)、五行、方位、对应数字。
- **64 卦表**:卦名、卦序、上下经卦、卦辞、《彖传》《象传》、所属八宫、卦宫五行。
- **384 爻 + 用九用六**:爻辞、爻位、当位与否、《小象传》。
- **干支 / 五行表**:十天干、十二地支、地支藏干、五行生克与刑冲合害。
- **纳甲表**:八宫卦的纳甲安爻规则(乾纳甲壬、坤纳乙癸……)。
- **六亲 / 六神**:以卦宫五行为"我",生克得六亲(父母/兄弟/子孙/妻财/官鬼);按日干起六神(青龙/朱雀/勾陈/螣蛇/白虎/玄武)。
- **二十四节气表**:用于由公历日期定**月建**与**日建**。

### 3.2 起卦模块(`Caster`)

| 方式 | 输入 | 产出 |
| --- | --- | --- |
| 铜钱摇卦 | 6 次三枚铜钱正反 | 6 爻的老阴/少阴/少阳/老阳 → 本卦 + 动爻 |
| 时间起卦 | 占问时间(年月日时干支) | 上下卦 + 动爻 |
| 数字报数 | 1~3 个数字 | 上下卦 + 动爻 |
| 随机 | 系统随机源 | 同摇卦 |

产出统一为:**本卦、动爻位置集合、变卦**。

### 3.3 排盘引擎(`Caster` → `Board`,以六爻为例)

确定性流水线(每一步都是规则映射):

1. **定卦宫与本卦/变卦**:由六爻阴阳定本卦,动爻变出变卦。
2. **纳甲装爻**:按八宫纳甲规则给每爻配天干地支 → 得每爻五行。
3. **定世应**:按"八宫世应"规则定世爻、应爻位置。
4. **配六亲**:以卦宫五行为"我",由各爻五行生克关系定六亲。
5. **起六神**:按占问日的日干,从初爻向上依次安六神。
6. **标动爻 / 变爻**:动爻及其变出之爻地支五行、化进 / 化退 / 化空等。
7. **定旬空**:按日柱旬定空亡地支。
8. **旺衰生克**:以**月建、日建**判各爻旺相休囚死,以及对用神的生扶克泄。
9. **取用神**:依问占事项类别(财/官/子孙/父母/兄弟)给出用神建议。

  "question": "近期求职是否顺利?",
  "questionCategory": "career",

              "ganzhi": { "year": "丙午", "month": "甲午", "day": "庚子", "hour": "庚辰" } },

### 3.5 测试策略
- 对静态数据表(64 卦、纳甲、六亲)做**金标准快照测试**。
- 对排盘流水线用**经典卦例**(古籍/教材案例)做端到端断言。
- 起卦算法的概率分布做统计测试(老阳/少阴比例)。

## 4. 解读层(LLM 的职责边界)

### 4.1 输入
- 预设工具层输出的**完整盘面 JSON**。
- 用户的问题文本 + 问占事项类别。
- (可选)用户画像/历史偏好,用于个性化表达。

### 4.2 角色与规则(System Prompt 要点)
- 设定为"资深六爻解卦师"。
- **硬约束**:只能基于传入盘面要素解读;**不得新增/修改任何卦象、地支、六亲、世应等数据**;吉凶判断必须**引用具体爻位 / 六亲 / 生克 / 旺衰**作为依据。
- 结构化输出:`整体断语 → 用神分析 → 关键爻 → 应期推测 → 建议 → 免责声明`。

### 4.3 RAG(检索增强)
- 向量库存储:64 卦 / 384 爻的**原文 + 白话注解 + 古籍断例**。
- 解读时按当前卦 / 动爻检索对应条目作为 grounding,降低幻觉、增强专业度与可信度。

### 4.4 多轮对话
- 首轮:基于盘面给整体断语。
- 后续:用户追问("大概什么时候应""对感情具体怎么讲"),模型在**同一盘面**上下文中延展,不重新起卦。
- 对话上下文中始终携带盘面 JSON(作为 system / tool 消息),保证不跑偏。

### 4.5 工程实现
- 使用 LLM 的 **工具调用 / JSON 模式** 约束输出结构。
- 后端做 **LLM 代理**,隐藏 API key、做限流与计费。
- 输出经**内容安全审核**后返回。

## 5. 客户端设计(iOS)

### 5.1 技术栈
- **SwiftUI** + Swift Concurrency(async/await)。
- `DivinationEngine` 作为本地 Swift Package,**离线**完成起卦排盘。
- **SwiftData**(或 Core Data)持久化历史卦例与对话。

### 5.2 主要界面
1. **起卦页**:选择方法(摇卦动画 / 时间 / 报数)、输入问题。
2. **排盘页**:六爻可视化(爻象、干支、六亲、六神、世应标记、动爻高亮、旬空标注),本卦/变卦切换。
3. **解读对话页**:首屏断语 + 可继续追问的聊天界面。
4. **历史 / 收藏**:按时间与事项分类回看。
5. **百科**:64 卦 / 384 爻原文与注解。
6. **每日一卦 / 推送**。

### 5.3 占卜动画设计

动画是这类 App 的"仪式感"核心,直接影响付费意愿。原则:**有仪式感、可跳过、不阻塞结果、适配无障碍**。整套占卜流程分阶段编排动画:

| 阶段 | 动画 | 交互 / 体感 | 技术实现 |
| --- | --- | --- | --- |
| ① 起卦准备 | 太极 / 八卦罗盘缓慢旋转,水墨晕染入场,引导用户"静心默念所问之事" | 输入问题 | SwiftUI `Canvas`/`TimelineView` + `phaseAnimator`,矢量罗盘用 Lottie |
| ② 摇卦 | 三枚铜钱在卦盘中翻滚、碰撞、落定;六次,逐爻**从下往上**生成 | **摇一摇**(CoreMotion)触发 + **触觉反馈**(CoreHaptics)模拟铜钱落地的"叮当" | 铜钱物理翻滚用 SpriteKit / 自绘 `Canvas`;落定用弹性动画 |
| ③ 成卦 | 六爻自下而上点亮:阳爻 `——` 连续笔,阴爻 `— —` 断开;水墨笔触从左到右"写出" | 逐爻有节奏地出现(stagger) | `matchedGeometryEffect` + 笔触遮罩动画(`mask` + `trim`) |
| ④ 动爻与变卦 | 动爻闪烁并标注 `○`(老阳)`×`(老阴);本卦 → 变卦的爻做**形变过渡**(阳⇄阴 morph) | 点击动爻可查看"为何动" | `withAnimation` + `matchedGeometryEffect` 做本卦/变卦切换 |
| ⑤ 排盘信息浮现 | 干支、五行、六亲、六神、世应、旬空逐项**淡入 + 轻微上浮**,世应用高亮描边 | 可点开每爻详情 | 标准 SwiftUI `transition` + `.animation` |
| ⑥ 解读 | LLM **流式打字机**输出 + "思考中"水墨涟漪 loading | 边出边读,可随时追问 | 流式 token 渲染 + 自定义 loading 视图 |

**美术风格**:水墨 / 国风(留白、笔触、印章红),罗盘与卦象用矢量保证清晰。

**技术选型小结**:
- **SwiftUI 原生**(`withAnimation`、`phaseAnimator`、`matchedGeometryEffect`、`Canvas`、`TimelineView`)能覆盖绝大多数:爻生成、信息浮现、变卦过渡、流式文字。
- **Lottie**(After Effects 导出 JSON)用于复杂矢量动画(罗盘旋转、太极、印章),便于美术独立产出、可控且包体小。
- **SpriteKit / 自绘 Canvas** 用于铜钱翻滚、粒子光效等需要物理感的部分。
- **CoreHaptics + CoreMotion** 增强"摇卦"的真实仪式感。

**性能与无障碍(必须)**:
- 动画与排盘计算**解耦**——引擎瞬时算完结果,动画只是"演出",任何时候允许**「跳过动画」**直达结果。
- 适配系统 **Reduce Motion**(减弱动态):降级为淡入淡出。
- 复杂粒子限制帧率与数量,低端机降级;首屏不被动画阻塞。

## 6. 后端设计

| 模块 | 职责 | 选型建议 |
| --- | --- | --- |
| API 网关 | 鉴权、限流、计费 | FastAPI / Node(Fastify) |
| LLM 代理 | 隐藏 key、流式转发、Prompt 组装 | 同上 |
| 向量检索 | 卦爻辞 / 注解 RAG | pgvector / Qdrant |
| 数据存储 | 用户、历史卦例、对话 | PostgreSQL |
| 内容安全 | 输入输出审核 | 国产合规审核 API |

> 排盘引擎**不在后端做**(放端上离线),后端只负责"解读 + 账号 + 知识库"。如需服务端复算/审计,可将同一份引擎逻辑以 Swift on Server 或独立服务复用。

### 6.1 排盘引擎:在线跑 vs 离线跑

引擎是确定性纯逻辑,放哪跑只影响**工程权衡**,不影响计算结果。两种方案对比:

| 维度 | 端上离线(推荐) | 服务端在线 |
| --- | --- | --- |
| **延迟** | 0 网络延迟,起卦排盘**瞬时**,动画体验顺滑 | 有网络往返,弱网/断网时卡顿或失败 |
| **可用性** | 无网也能起卦排盘(只有解读才需联网) | 完全依赖网络 |
| **隐私/合规** | 占问内容、盘面**不出设备**,隐私最好 | 占问内容上云,需处理隐私与合规 |
| **成本** | 无服务器算力成本 | 每次请求消耗服务器资源 |
| **可更新性** | 引擎逻辑随 App 下发,修 bug / 改规则**需发版**(走 App Store 审核) | **热更新**,改完即生效,无需发版 |
| **集中治理** | 难做统一统计 / A-B / 审计 | 逻辑与数据集中,便于统计、审计、灰度 |
| **包体积** | 静态数据表(64 卦/纳甲表等)计入安装包 | 数据在服务端,安装包小 |

**结论与折中方案(本项目采用)**:
- **排盘引擎放端上离线**——确定、快、隐私好,且术数规则稳定、极少变更,"需发版才能改"不是大问题。
- **解读必须在线**(LLM 在服务端代理,隐藏 key)。
- **两个折中点弥补离线的短板**:
  1. **数据与代码分离**:把 64 卦/爻辞/注解等**静态数据表做成可远程增量更新**(下发 JSON/数据包),这样修订文案、补数据无需发版,只有"排盘算法"本身改动才发版。
  2. **保留服务端复算接口**:用同一份引擎逻辑在服务端提供一个校验/审计端点,用于疑难排查、客服核对、或未来需要服务端统计时复用——平时不走它。

> 一句话:**算在端上(离线、快、私密),数据可云端更新,解读在云端(LLM)。**

### 10.2 M1 — 排盘引擎(核心,~3 周)

- [ ] **静态数据资产**:八卦、64 卦表(卦名/卦序/上下卦/卦辞/彖象/卦宫五行)、384 爻辞、干支五行表、地支藏干、纳甲表、六亲规则、六神规则、二十四节气表。以结构化资源文件(JSON)内置。

## 11. 关键风险与对策

| 风险 | 对策 |
| --- | --- |
| LLM 在术数计算上幻觉 | 计算全部交给确定性引擎,LLM 只解读不计算(本设计核心) |
| 排盘规则实现错误 | 静态表快照测试 + 经典卦例端到端测试 |
| 合规 / 内容风险 | 国产合规模型 + 内容审核 + 全局免责声明 |
| 成本(LLM 调用) | 端上离线排盘省算力;解读做缓存/分层计费 |
